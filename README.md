# 面试准备：从 JD 到高命中率答案

> 大多数人的面试准备，80% 时间花在了不会考的地方。
> 这套方法论让你在只有 JD 的情况下，把准备时间精准分配到面试官真正要验证的能力上。

## 核心理念

面试不是答题比赛。面试官不是在找「谁准备得最多」，是在找**谁解了他的题**。

你的题 = JD 上排在前面的那几条。后面的都是背景板。

## 四步法

### 第一步：JD 权重解码（2 分钟）

拿到 JD，不要从头读到尾。先标序号：

```
JD 第1条：[_______________] ← 占面试50-60%时间
JD 第2条：[_______________] ← 占20-30%
JD 第3条：[_______________] ← 占10-15%
JD 第4条及以后：[_________] ← 可能完全不问
```

**规则：** 你的准备时间分配必须按这个比例。如果准备 2 小时，第一条花 1 小时，第二条花 40 分钟，第三条花 20 分钟。第四条以后，准备一个能说的亮点就行。

**这个规则从哪来的：** 我在多次面试后复盘了一个规律——面试官手里有评估表，但人的注意力是有限的。前 20 分钟他会深挖最关心的那一条。挖透了，才开始看第二、第三条。而一场面试通常 30-45 分钟，第四条以后根本没时间问。

### 第二步：核心题预判（5 分钟）

JD 第一条 = 面试官最想验证的能力。但他不会直接问「你做过这个吗」——他会问三个层次：

**第一层：验证你做过**
- 「讲讲你在 XX 方面的经验」
- 准备一个 2 分钟的完整案例，包含背景、你的角色、具体动作、结果

**第二层：验证你理解**
- 「你做的时候遇到了什么问题？怎么解决的？」
- 准备一个你克服的具体困难，不要讲「团队协作问题」这种万能的

**第三层：验证你能迁移**
- 「如果来我们这里，同样的方法论你会怎么用？」
- 准备一个从旧公司到新公司的迁移思路

**如果 JD 前三项都跑这个三问，你总共需要准备 9 个回答片段。** 这比准备 30 道题有效得多，因为 80% 的面试问题都是这三层的变体。

### 第三步：提问即信号（面试中实时使用）

面试官问你的每一个问题，都对应 JD 上的某一条。反推：

| 面试官问 | 他在验证 JD 的第几条 | 他想听到什么 |
|---------|-------------------|-------------|
| 「讲讲你做过的最成功的项目」 | 第1条（核心能力） | 你有完整 ownership，不只是参与者 |
| 「团队多少人？你怎么分工的？」 | 第2条（leadership/协作） | 你不是单打独斗，你能带人 |
| 「数据是怎么来的？准不准？」 | 第3条（数据/分析能力） | 你不拍脑袋，你有数据意识 |

**如果面试官没问你某个能力，说明这条在 JD 上并不重要——不管 JD 写得多么花哨。**

### 第四步：被纠正 = 还在局里

面试官纠正你、打断你、把你拉回来——这不是坏信号。这是最好被低估的信号：

- **他在乎**：如果他不在乎，他会让你讲完然后心里标记「跑偏了」
- **他还在投入**：纠正说明他还在跟你互动，没有放弃这场面试
- **他在给机会**：拉回来是让你重新对上他的频道

**被纠正时的正确回应：** 停下来，承认偏了，重新对准。不要解释、不要绕回、不要说「殊途同归」。

## 使用方式

### 快速版：面试前 15 分钟

把这篇文章的「第一步」和「第二步」跑一遍——2 分钟标序号，5 分钟预判核心题，8 分钟准备回答片段。

### 完整版：面试前 2 小时

跑完四步，再套用 `examples/` 里的八个案例，看同样的方法论在不同行业、不同岗位、不同阶段公司是怎么生效的。

### Skill 版：让 AI 工具帮你跑

把 `template/interview-prep.template.md` 的内容复制到你的 AI 编程工具的 instructions 里，然后把 JD 贴给 AI——它会自动跑四步法，输出结构化准备文档。
